# Realtime Updates

> See also: [Architecture](architecture.md) · [Configuration](configuration.md) · [REST API](api.md)

## Goal

SoManAgent uses a centralized realtime stack to refresh the frontend when backend state changes.

The current standard transport is **Mercure**:
- the backend publishes normalized application events to a hub
- the frontend subscribes to canonical topics through one shared client
- screens react by invalidating or refreshing their local data stores

This replaces screen-specific SSE endpoints and keeps realtime concerns reusable across the application.

## Architecture

The backend keeps a hexagonal split:
- application services publish business events through `RealtimeUpdateService`
- `RealtimeUpdateService` depends on the `RealtimePublisherPort`
- `MercureRealtimePublisher` is the current adapter implementing that port
- `RealtimeTopicFactory` centralizes topic conventions

The frontend also keeps one shared entry point:
- `realtimeClient` owns the Mercure `EventSource` integration
- `realtimeTopics` centralizes topic naming
- page-specific hooks subscribe through this client and invalidate the relevant React Query caches

## Event Envelope

Every Mercure message uses the same JSON envelope:

```json
{
  "id": "uuid-v7",
  "type": "task.changed",
  "occurredAt": "2026-04-05T12:34:56+00:00",
  "payload": {
    "projectId": "…",
    "ticketId": "…",
    "taskId": "…",
    "reason": "validated"
  }
}
```

Rules:
- `id` is generated by the backend and forwarded as the Mercure update ID
- `type` is stable and screen-agnostic
- `payload` contains business identifiers and optional contextual fields
- the frontend must react to `type` and `payload`, not to topic names alone

## Topic Conventions

Topics are canonical URIs under `https://somanagent.local/realtime`.

Current conventions:
- project root: `https://somanagent.local/realtime/projects/{projectId}`
- project tickets: `https://somanagent.local/realtime/projects/{projectId}/tickets`
- one ticket: `https://somanagent.local/realtime/projects/{projectId}/tickets/{ticketId}`
- one task: `https://somanagent.local/realtime/projects/{projectId}/tasks/{taskId}`
- project audit: `https://somanagent.local/realtime/projects/{projectId}/audit`
- project tokens: `https://somanagent.local/realtime/projects/{projectId}/tokens`

Rules:
- do not encode screen names in topics
- prefer entity or aggregate scopes
- publish to every topic needed by legitimate subscribers
- keep the topic factory as the source of truth in both backend and frontend

## Current Published Event Types

- `project.changed`
- `ticket.changed`
- `ticket.deleted`
- `task.changed`
- `task.deleted`
- `ticket.log.changed`
- `execution.changed`

New event types must remain business-oriented and reusable.

## Infra Topology

- the Mercure hub runs as a dedicated Docker service
- the backend publishes to `MERCURE_PUBLISH_URL`
- the browser subscribes through `/.well-known/mercure`
- Vite proxies this path in development
- Nginx proxies the same path for backend-served environments

This keeps the browser-side subscription URL stable while preserving an independent hub service.

## Reconnection and Missed Events

Mercure handles browser reconnection through `EventSource`.

The hub is configured with Bolt persistence so subscribers can resume from the last event ID automatically after temporary disconnections.

Because of this:
- backend events must always provide a stable update ID
- frontend code should avoid rebuilding subscriptions unnecessarily
- cache invalidation must remain idempotent

## Extension Guidelines

When adding a new realtime use case:
- add or reuse canonical topics in the topic factories first
- extend `RealtimeUpdateService` with a business-oriented publish method
- keep Mercure-specific concerns inside the adapter
- consume updates from the shared frontend client instead of instantiating `EventSource` directly in screens
- document any new topic family or event type in this file
